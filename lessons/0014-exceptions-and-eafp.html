Exceptions & EAFP

Most languages treat exceptions as a fire alarm. Python treats them as a normal way to ask "did that work?" — and embracing that makes your code shorter, faster, and more correct.

You already know how to throw and catch. The delta here is attitude: in Python you reach for exceptions early and often, you let the failure tell you what went wrong, and you stop pre-checking conditions the language will check for you anyway.

EAFP vs LBYL

Python has a name for its preferred style: EAFP — "easier to ask forgiveness than permission." The alternative, common elsewhere, is LBYL — "look before you leap," guarding every operation with a precondition check.

LBYL — look before you leap

# check, then use
if key in d:
    value = d[key]
else:
    value = default

EAFP — ask forgiveness

# just try it
try:
    value = d[key]
except KeyError:
    value = default

The EAFP version does the lookup once instead of twice, and it has no gap where the key could vanish between the in check and the index (a real concern with shared or concurrent state). For this exact case, the truly idiomatic answer is even shorter:

value = d.get(key, default)
The mental shift: EAFP is not "skip error handling." It's "let the operation itself be the test." You don't ask whether a file exists and then open it — you open it and catch FileNotFoundError, because the answer to "can I open it?" is the open.

The full statement: try / except / else / finally

try:
    f = open(path)             # risky work
except FileNotFoundError:
    log("missing")          # handle a specific failure
else:
    data = f.read()           # runs ONLY if try raised nothing
finally:
    cleanup()                 # always runs, success or not

Each clause has one job. except handles a failure; else holds the success-only code you want to keep out of the try (so it can't accidentally catch a second, unrelated error); finally runs no matter what — return, raise, or fall through — which makes it the home for cleanup.

Catch specific exceptions — never bare except

Hides bugs

try:
    risky()
except:            # catches EVERYTHING
    pass

Says what you mean

try:
    risky()
except (KeyError, IndexError) as e:
    log(e)

A bare except: also swallows KeyboardInterrupt and SystemExit, so Ctrl-C stops working and genuine bugs disappear silently. Name the exceptions you actually expect; group related ones in a tuple, and bind the instance with as e when you need its details.

Raising, re-raising, and chaining

# signal an error yourself
raise ValueError("age must be non-negative")

# re-raise the current exception unchanged (keeps its traceback)
try:
    parse(raw)
except ValueError:
    log("bad input")
    raise

# wrap a low-level error in a domain one, preserving the cause
try:
    cfg = json.loads(text)
except json.JSONDecodeError as err:
    raise ConfigError("could not load config") from err

A bare raise inside an except re-throws the exact exception being handled, traceback intact — better than raise e. The raise X from err form sets the cause, so the traceback shows both your error and the one underneath it ("The above exception was the direct cause...").

Your own exception types

A custom exception is just a class that subclasses Exception — usually the whole body is pass. The name is the value:

class ConfigError(Exception):
    """Raised when configuration cannot be loaded."""

Callers can then write except ConfigError to catch your errors specifically, without accidentally grabbing unrelated ValueErrors. Subclass a more specific built-in (like ValueError) only when your error genuinely is that kind of error.

Exceptions as control flow

Here's the cultural difference. In many languages exceptions are "exceptional" — reserved for failures. Python uses them for ordinary control flow too. The clearest example: every for loop is built on an exception. When an iterator is exhausted, it raises StopIteration, and the loop catches it to know when to stop.

So don't flinch at a try on a hot path the way you might elsewhere. When the success case dominates, EAFP is often faster than LBYL, because the happy path pays no checking cost at all.

Ignoring an error on purpose: contextlib.suppress

from contextlib import suppress

with suppress(FileNotFoundError):
    os.remove(path)        # gone already? fine.

When you genuinely want to do nothing on a specific error, this reads better than a try/except .../pass — it names the intent and the suppressed type on one line. (It ties back to the context-manager lesson: suppress is just a context manager.)
