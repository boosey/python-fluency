Welcome to lesson fifteen: type hints. If you come from a statically typed language, the first thing to unlearn is the biggest surprise of Python's hints. They are not enforced at runtime; the interpreter reads them, then completely ignores them. A function annotated to take a string will happily run when you pass it an integer, and Python will not raise a type error. Hints exist for your tools, your editor, and the next human to read the code, not for the runtime. A type checker like mypy or pyright reads them and flags mismatches before you ever run the program. The basic shape is simple: write a colon and a type after a parameter, and an arrow and a type for the return. So define greet, name colon str, arrow str, and the contract is now visible to every reader and every tool. You annotate variables the same way, with a colon and a type after the name. Since Python 3.9 you write container types with the built-in names directly: list of int, dict of str to int, tuple of int and ellipsis. The old capital-L List and capital-D Dict from the typing module still work, but they are deprecated in favour of the builtins. For a value that might be missing, write X pipe None, which is the modern spelling of Optional. The same pipe builds any union, so int pipe str means either an integer or a string. From the typing module you still import the things that have no builtin form, like Any, Callable, and Iterable. For duck typing there is Protocol, which says any object with these methods fits, regardless of its base class. That is structural typing, and it matches Python's culture: if it quacks like a duck, the checker accepts it. Dataclasses lean on annotations too; the field types you wrote in lesson thirteen are exactly these hints. The idiom is to annotate your public function signatures, where the contract matters most. Resist annotating every obvious local variable; a count set to zero does not need to be told it is an int. Lesson 15 · Idiomatic Python

Type hints

Annotations that document your intent and power your editor and type checker — without changing how a single line runs.

You already know static types from other languages. The delta here is small but surprising: Python's hints look like static types, yet they're advisory. The win is learning to read and write modern annotations so tooling can catch your mistakes before you run the code.

The surprise: hints are not enforced

The mental shift: annotations are not checked at runtime. The interpreter parses them and stores them, then ignores them when the function runs. They're for tooling (mypy, pyright), your editor, and human readers — not for the runtime.
def greet(name: str) -> str:
    return "hi " + name

greet(42)  # runs! TypeError only from the + , not from the hint

A type checker reads name: str and flags the call as wrong before you run it. Python itself does no such check — that's the part that trips up newcomers from static languages.

Annotating functions and variables

A colon and a type after a parameter; an arrow and a type for the return. The same colon form annotates a variable.

def scale(values: list[int], factor: int) -> list[int]:
    return [v * factor for v in values]

count: int = 0
name: str = "ada"

Built-in generics (3.9+)

Write container types with the lowercase builtins directly — no import needed.

nums:    list[int]
lookup:  dict[str, int]
pair:    tuple[int, str]       # exactly two items
row:     tuple[int, ...]       # any number of ints

Legacy (deprecated)

from typing import List, Dict

xs: List[int]
m:  Dict[str, int]

Modern (3.9+)

# no import at all

xs: list[int]
m:  dict[str, int]

Optional and unions with |

Since 3.10, write X | None for a value that might be missing — this is the modern Optional[X]. The same pipe builds any union.

def find(key: str) -> str | None:        # may return a str or None
    return _cache.get(key)

def parse(raw: str) -> int | float:        # either numeric type
    ...

The older spelling Optional[str] (from typing) is identical to str | None; prefer the pipe in new code on 3.10+.

What still comes from typing

Builtins cover containers, but some helpers have no lowercase form. Import those from typing:

from typing import Any, Callable, Iterable, Optional

handler: Callable[[int, str], bool]   # takes (int, str), returns bool
items:   Iterable[str]                # anything you can iterate
data:    Any                          # opt out of checking

List, Dict, Tuple from typing still exist but are deprecated — reach for the builtins.

Duck typing, made checkable: Protocol

Python's culture is "if it quacks like a duck." Protocol turns that into something a checker understands: any object with the right methods fits — no inheritance required.

from typing import Protocol

class Sized(Protocol):
    def __len__(self) -> int: ...

def describe(x: Sized) -> str:
    return f"has {len(x)} items"   # list, str, dict... all fit structurally
Tie-in: dataclasses (lesson 13) are these annotations — the field types you wrote are hints the checker reads. And from __future__ import annotations at the top of a file makes all annotations lazy strings, handy for forward references; you'll see it in older or self-referential code.

The idiom: hint the boundaries

Annotate public function signatures, where the contract matters and tools help most. Don't over-annotate obvious locals — inference already knows count = 0 is an int.

# good: the signature documents the contract
def load(path: str) -> dict[str, int]:
    total = 0                 # no hint needed — obvious
    ...
