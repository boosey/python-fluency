Welcome to lesson five: iteration and its built-ins. Here is the headline that trips up almost every newcomer: Python has no C-style index loop. There is no for-i-equals-zero, i-less-than-length, i-plus-plus. Instead you iterate the object itself, directly, with for x in xs. So the first reflex to unlearn is for i in range(len(xs)) followed by xs at i. If you only need the items, just say for x in xs and reach for the value by name. When you genuinely need the index too, wrap the iterable in enumerate. Enumerate yields index-value pairs, and you can pass start equals one when you want human-friendly numbering. To walk two sequences in lockstep, use zip, which pairs up their items and stops at the shorter one. A lovely trick: zip star matrix transposes a matrix, turning rows into columns. In Python three-ten and later, zip strict equals True raises if the inputs differ in length, catching silent bugs. Python also has a control structure most languages lack entirely: for-else. The else clause after a loop runs only if the loop finished without hitting a break, which is perfect for search-then-not-found logic. When you want to walk backwards, reach for reversed rather than fiddling with negative indices. When you want a different order, sorted takes a key function and a reverse flag, and returns a new list. Iterating a dict gives you its keys by default; use dot items for key-value pairs, dot values for just the values. Under all of this sits the iterator protocol: iter gives you an iterator, next pulls the following value, and StopIteration signals the end. You rarely call those by hand, but knowing they exist explains why generators, which we meet later, fit so neatly. Iteration & its built-ins

Python has no C-style index loop. You iterate objects directly — and a handful of built-ins make almost every "I need the index too" case index-free.

You know how to loop. The win here is narrow and concrete: never reach for range(len(xs)) again. Iterate the thing itself, and let enumerate, zip, and friends carry the bookkeeping.

The loop Python doesn't have

Most languages

// index, bound, increment
for (i = 0; i < xs.length; i++) {
    use(xs[i]);
}

Idiomatic Python

# iterate the object directly
for x in xs:
    use(x)

The first reflex to retire is for i in range(len(xs)): followed by xs[i]. If you only need the items, name them: for x in xs. The index was never the point — the value was.

When you really need the index: enumerate

# index AND item, no manual counter
for i, x in enumerate(xs):
    print(i, x)

# human-friendly numbering starts at 1
for rank, name in enumerate(winners, start=1):
    print(f"{rank}. {name}")

enumerate(xs) yields (index, item) pairs, which you unpack right in the for header. Pass start=1 when the number is for people, not for indexing.

The mental shift: the index is a side product of iteration, not the driver of it. You ask for it explicitly with enumerate when you need it, and otherwise it stays out of your way.

Two sequences at once: zip

# walk in parallel; stops at the shorter input
for name, score in zip(names, scores):
    print(name, score)

# transpose a matrix: rows become columns
rows = [[1, 2, 3], [4, 5, 6]]
cols = list(zip(*rows))   # [(1, 4), (2, 5), (3, 6)]

zip pairs items position-by-position and stops at the shortest argument. The zip(*rows) idiom spreads the rows as separate arguments, so column k is gathered from every row — a one-line transpose.

Python 3.10+: pass zip(a, b, strict=True) to raise a ValueError when the inputs differ in length, instead of silently truncating to the shortest. Reach for it when "same length" is an invariant you want enforced.

A loop clause most languages lack: for … else

# the else runs only if the loop did NOT break
for item in haystack:
    if matches(item):
        found = item
        break
else:
    found = None   # searched everything, no break ever fired

The else attached to a for (or while) runs only if the loop completed without hitting break. Read it as "for … else if-nothing-broke". It's the clean home for the searched-and-found-nothing branch.

Order, on demand: reversed and sorted

# walk backwards without negative-index gymnastics
for x in reversed(xs):
    ...

# sorted returns a NEW list; key picks what to sort on
by_len  = sorted(words, key=len)
by_high = sorted(scores, reverse=True)
people  = sorted(people, key=lambda p: p.age)

sorted never mutates its argument — it returns a fresh list, so it composes inside expressions. key takes a function applied to each item to decide the order; reverse=True flips it.

Iterating a dict

# iterating a dict gives its KEYS by default
for key in prices:
    ...

for key, value in prices.items():   # pairs
    ...
for value in prices.values():     # just values
    ...
for key   in prices.keys():       # explicit keys
    ...

A bare for k in d walks the keys. When you want both sides, .items() is the idiom — never index back into the dict with d[k] just to fetch the value you already have.

Underneath: the iterator protocol

All of this rests on a tiny contract. iter(obj) returns an iterator; next(it) pulls the following value; when there are none left it raises StopIteration, which the for loop catches for you:

it = iter([10, 20])
next(it)   # 10
next(it)   # 20
next(it)   # raises StopIteration

You rarely call these by hand, but they explain why a for loop works on anything iterable — and they're exactly the hooks that generators (a later lesson) implement to produce values lazily.

Lazy by default: range doesn't build a list — it computes numbers on demand as you iterate. That's why range(1_000_000) costs almost nothing until you walk it.
