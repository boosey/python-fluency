Welcome to lesson sixteen: structural pattern matching. If you came from C, Java, or JavaScript, the words match and case will make you think switch — and that instinct will mislead you. A C switch compares one value against a list of constants and falls through unless you break. Python's match does something different: it matches the subject against the shape of the data, and binds names as it goes. So the real win here is reading match statements with confidence, knowing they destructure, not just compare. The simplest cases do look switch-like: case literal, with case underscore as the catch-all wildcard. But the moment you write case Point of x and y, you are no longer comparing — you are pulling x and y out of the object. Sequence patterns work the same way: case open-bracket a comma b matches a two-element sequence and binds a and b. Add a star, as in case first comma star rest, and rest captures everything after the first element, just like unpacking. Mapping patterns match dictionaries by the keys you name, ignoring any extra keys the dict happens to carry. Class patterns can use positional arguments or keyword arguments, so case Point with x equals zero and y equals zero matches only the origin. You can add a guard: case a comma b, then if a equals b, runs the case only when that extra condition also holds. And you can combine alternatives with the pipe: case four-oh-one or four-oh-three or four-oh-four matches any of those three. Now the single most important gotcha, the one that trips up everyone. A bare lowercase name in a case does not compare against a variable of that name — it captures. So case red will match anything and bind it to red, even if you meant to check against a constant called red. To match against a constant, you must use a dotted name, like Color dot RED, or an attribute lookup. That one rule explains most of the confusion people have with match, so hold on to it. Reach for match when you are branching on the shape of structured data: parsed JSON, an abstract syntax tree, a command, an event. Structural pattern matching

It looks like a switch, but it isn't one. match branches on the shape of your data and pulls values out as it goes — Python 3.10's most powerful reading skill.

You'll meet match in parsers, event handlers, and anything that walks structured data. The goal of this lesson is to read match/case fluently — and to internalise the one rule that confuses everyone.

Not a switch — it destructures

What you expect (a switch)

// compare one value to constants
switch (status) {
  case 404: ...; break;
  default: ...;
}

What match actually does

# match the SHAPE, bind as you go
match point:
    case Point(x, y):
        # x and y are now bound
        ...

A switch asks “is this value equal to that constant?”. match asks “does this data have this structure?” — and where it does, it binds names to the pieces. There is no fall-through and no break; the first matching case wins.

Literals and the wildcard

The switch-like layer: match literal values, with case _ as the catch-all. The underscore is the wildcard — it matches anything and binds nothing.

match command:
    case "quit":
        ...
    case "help":
        ...
    case _:
        # anything else
        ...

Capture & sequence patterns

A name inside a pattern captures the matched value. Sequence patterns destructure lists and tuples — including a starred name for “the rest”, exactly like unpacking assignment:

match coords:
    case []:
        # empty sequence
        ...
    case [a, b]:
        # exactly two items → a, b bound
        ...
    case [first, *rest]:
        # first item, then rest = everything after
        ...

Mapping & class patterns

Mapping patterns match dictionaries by the keys you name and ignore extra keys. Class patterns match an instance and pull out attributes, positionally or by keyword:

match shape:
    case {"type": "circle", "r": r}:
        # r captured; other keys ignored
        area = 3.14159 * r * r
    case Point(x=0, y=0):
        # only the origin matches
        ...
    case Point(x=0, y=y):
        # on the y-axis; y captured
        ...

Positional class patterns like case Point(x, y) rely on the class declaring __match_args__ (dataclasses do this for you); keyword patterns like Point(x=0, y=0) always work.

Guards and OR patterns

A guard is a trailing if that adds a condition the pattern alone can't express. A | (OR) pattern matches any of several alternatives:

match pair:
    case [a, b] if a == b:
        # two equal items
        ...

match status:
    case 401 | 403 | 404:
        # any of these three
        ...

The one rule everyone trips on

A bare lowercase name in a case is a capture, not a comparison. case red: does not check whether the subject equals a variable named red — it matches anything and binds it to red (and so acts like a catch-all, shadowing later cases).

To match against a constant, use a dotted name — an attribute lookup the parser recognises as a value, not a binding:

The trap (always matches)

match color:
    case RED:
        # captures! binds RED to color
        ...

The fix (compares by value)

match color:
    case Color.RED:
        # dotted → matches the constant
        ...

Literals (404, "quit") and dotted names (Color.RED, math.pi) are value patterns. Bare names are always captures. When in doubt, qualify it.

When to reach for it

Use match when you branch on the structure of data: parsed JSON, an AST node, a command, an event object. For a plain equality check on a single value, an if/elif chain or a dict lookup is still simpler and clearer.
