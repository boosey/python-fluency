Welcome to lesson thirteen: dataclasses and named tuples. This lesson is about modelling plain data without writing a wall of boilerplate. In most languages, a small data-holding class means a constructor, a field list, an equals method, and a toString — all by hand. Python generates that whole chassis for you from a single decorator and a few annotated fields. Put @dataclass above a class, list each field as a name and a type, and you are done. The decorator writes __init__, __repr__, and __eq__ for you, derived directly from those annotations. Remember the dunder lesson: __repr__ controls how the object prints, and __eq__ controls when two objects count as equal. A dataclass wires both up automatically, so two points with the same coordinates compare equal out of the box. Fields can have defaults, written just like default arguments to a function. But never default a field to a mutable like an empty list, because that list would be shared across every instance. Instead use field with default_factory set to list, and each instance gets its own fresh list. That is the same mutable-default trap from lesson eight, solved the dataclass way. Pass frozen equals True and instances become read-only and hashable, so they can live in sets and dictionary keys. Pass order equals True and Python generates the comparison operators, so you can sort instances by their fields. A dataclass is still an ordinary class, so you can add methods and computed properties as normal. If you need to derive a field after init, define __post_init__ and finish the setup there. For a lightweight immutable record, typing.NamedTuple gives you a tuple that also has named fields. The older collections.namedtuple does the same thing with a different, string-based syntax; reach for the typing form in new code. Pick a NamedTuple when the value is a fixed, ordered record you might unpack like a tuple; pick a dataclass when it has behaviour or mutates. And remember: a plain dict or tuple is often enough — a dataclass earns its keep once the shape is fixed and you want a real type. Lesson 13 · Idiomatic Python

Dataclasses & named tuples

Model plain data without the ceremony. One decorator writes the constructor, the printer, and the equality check — so you describe the shape and Python builds the machinery.

You already know how to write a class that just holds a few fields. The point of this lesson isn't that you can — it's that idiomatic Python asks you to declare the fields and lets the language generate the boilerplate.

The boilerplate Python replaces

Most languages (and old Python)

class Point:
    def __init__(self, x, y):
        self.x = x
        self.y = y
    def __repr__(self):
        return f"Point(x={self.x}, y={self.y})"
    def __eq__(self, other):
        return (self.x, self.y) == (other.x, other.y)

Idiomatic Python

from dataclasses import dataclass

@dataclass
class Point:
    x: int
    y: int

Both define the same class. The @dataclass decorator reads the annotated fields and generates __init__, __repr__, and __eq__ for you — exactly the dunders from the earlier lesson. So Point(1, 2) prints as Point(x=1, y=2), and Point(1, 2) == Point(1, 2) is True without you writing a line of it.

The annotations are mandatory: a field is any class-level name with a type annotation. x: int declares a field; x = 0 with no annotation is not a dataclass field, just a plain class attribute. The type is a hint — Python doesn't enforce it.

Defaults — and the mutable-default trap

Fields take defaults just like function arguments, with the same rule that defaulted fields must follow non-defaulted ones:

@dataclass
class User:
    name: str
    active: bool = True

But the mutable-default trap from lesson 08 still bites: a field defaulted to [] would be shared by every instance. Dataclasses detect this and make you spell out a per-instance factory instead:

from dataclasses import dataclass, field

@dataclass
class Cart:
    owner: str
    items: list = field(default_factory=list)  # fresh list each time

field(default_factory=list) calls list() once per instance, so every Cart gets its own list. Writing items: list = [] is an error a dataclass refuses to let you make.

frozen and order — immutability and sorting

@dataclass(frozen=True, order=True)
class Version:
    major: int
    minor: int

frozen=True makes instances read-only: assigning to a field raises, and because they can't change, Python also generates __hash__ — so frozen instances work as set members and dict keys. order=True generates <, <=, >, >= by comparing fields as a tuple, top to bottom — so sorted(versions) just works.

Still a normal class: a dataclass can have methods, properties, and class attributes like any other. If a field must be computed from the others, set it in __post_init__, which runs right after the generated __init__ finishes.

Lightweight records: NamedTuple

When the value is a small, fixed, immutable record — and you'd happily unpack it like a tuple — a named tuple is lighter than a dataclass. The modern spelling is typing.NamedTuple, which reads just like a dataclass:

from typing import NamedTuple

class Point(NamedTuple):
    x: int
    y: int

p = Point(1, 2)
p.x          # 1  — access by name
x, y = p     # also unpacks like a tuple

The older collections.namedtuple does the same thing with a string-based call — Point = namedtuple("Point", ["x", "y"]). It still works everywhere, but in new code prefer the typing.NamedTuple class form: it's annotated, supports defaults and methods, and reads better.

Which one, and when none?

  • NamedTuple — a fixed, ordered, immutable record you might unpack or compare as a tuple (coordinates, an RGB colour, a return value).
  • dataclass — the shape has behaviour, mutates, or you want frozen/order/__post_init__ control.
  • plain dict or tuple — the keys are dynamic or one-off and no fixed type earns its keep; reach for a class once the shape is settled.
