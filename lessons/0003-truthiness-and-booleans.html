Welcome to lesson three: truthiness and boolean idioms. This is the lesson that makes your conditionals stop looking ported and start looking native. In Python, every object already knows whether it is truthy or falsy, so you rarely have to ask explicitly. Empty containers, the number zero, an empty string, and None are all falsy; anything non-empty is truthy. That means you write if items, not if len of items is greater than zero. And you write if not items to ask whether a list is empty, never items equals empty brackets. The conditional reads like plain English, and that is the whole point. Now the most important rule about None: always compare it with is and is not, never with double equals. None is a singleton, so is checks identity, which is exact, fast, and what PEP 8 asks for. Next, the or operator is the classic way to supply a default: name equals name or anon. But watch the trap: if zero or an empty string is a valid value, or will wrongly replace it, because those are falsy. When zero and empty string are legitimate, reach for x if x is not None else default instead. Here is the part that surprises people from other languages: and and or do not return booleans. They short-circuit and return one of the actual operands, so a or b can hand you a or it can hand you b. For aggregate questions, any and all read beautifully with a generator expression flowing through them. any of this for that gives you True the moment one item qualifies, and all the moment one fails. Python also lets you chain comparisons, so zero is less than or equal to x less than ten is one clean expression. And membership tests read like a sentence: if x in collection, or if key in dict. One last thing: never write equals True or equals False; the value is already a boolean, so just use it. And a small curiosity for later: bool is a subclass of int, so True really does behave like one. Scroll down, take the quick checks, and replay this narration whenever you like. Lesson 03 · Idiomatic Python

Truthiness & boolean idioms

Python objects already know if they're "empty" — lean on that and your conditionals read like English instead of like a port from another language.

You know how to write an if. The win here is smaller and sharper: the handful of boolean idioms that idiomatic Python leans on everywhere, so you read and write them without a second thought.

Truthiness: let the object answer

Most languages

# ask explicitly
if len(items) > 0:
    ...
if items == []:
    ...

Idiomatic Python

# the object is already truthy/falsy
if items:
    ...
if not items:
    ...

Empty containers, 0, "", and None are all falsy; anything non-empty is truthy. So if items: means "if there's anything in items," and if not items: means "if it's empty." No len, no comparison to a literal.

is None, never == None

# identity check — what PEP 8 asks for
if value is None:
    ...
if value is not None:
    ...

None is a singleton — there is exactly one of it — so you compare with is/is not, which tests identity. It's exact, fast, and can't be fooled by an object that overrides ==.

The mental shift: "is it empty?" and "is it None?" are different questions. if not value: is true for None, 0, "", and [] alike. When you specifically mean "was nothing passed?", say is None.

or for defaults — and its trap

# common, concise default
name = name or "anon"

# trap: 0 and "" are valid but falsy, so `or` replaces them
count = count or 10      # count == 0 becomes 10 — bug!

# safe when 0 / "" are legitimate values
count = count if count is not None else 10

name or "anon" is great when any falsy input should fall back. But if 0 or "" are real values you must keep, use the explicit x if x is not None else default form instead.

and/or return operands, not booleans

# short-circuit: the result is one of the actual operands
"hi" or "bye"     # => "hi"   (first truthy)
"" or "bye"       # => "bye"  (first was falsy)
"hi" and "bye"    # => "bye"  (both truthy, returns last)
0 and "bye"       # => 0     (stops at first falsy)

This is exactly why name or "anon" works: or returns the first truthy operand, or the last one if none qualify. The value you get back is the operand itself, not a coerced True/False.

any / all with a generator

# True as soon as one item qualifies — stops early
if any(n < 0 for n in numbers):
    ...

# True only if every item qualifies
if all(isinstance(x, str) for x in values):
    ...

Feed any and all a generator expression and you get a readable, short-circuiting aggregate: any stops at the first match, all stops at the first failure. No accumulator flag required.

Chained comparisons & membership

# chain comparisons like math — x is evaluated once
if 0 <= x < 10:
    ...

# membership reads like a sentence
if x in collection:
    ...
if key in some_dict:      # tests keys, not values
    ...

0 <= x < 10 means exactly what it says and is the same as 0 <= x and x < 10 — but x is evaluated only once. And in is the idiomatic membership test; on a dict it checks the keys.

Don't write == True. If flag is already a boolean, if flag: says it best; if flag == True: is noise. (Aside: bool is a subclass of int, so True == 1 — another reason to compare by truthiness, not by equality.)
