Welcome to lesson ten: the data model and dunder methods. Here is the big idea that makes Python feel coherent once it clicks. Python has no keyword for operator overloading and no special interface you declare. Instead, the language defines a set of named methods, written with double underscores, that the syntax quietly calls for you. We call these dunder methods, short for double underscore. When you write len of x, Python calls x dot dunder len. When you write a plus b, Python calls a dot dunder add of b. The builtins already implement these methods, which is why lists, strings, and dicts feel native. Your own classes opt in by implementing the very same protocol. Let us build one small class, a two dimensional vector, and make it feel built in. First, dunder repr: it returns an unambiguous, developer facing string, ideally one you could paste back to recreate the object. Dunder str is the friendly version for end users, and if you skip it, str falls back to repr. So implement repr first; it is the one that helps you in the debugger and the REPL. Next, dunder eq gives you value equality, so two vectors with the same components compare equal. If you define dunder eq and want the object to stay hashable for sets and dict keys, you must also define dunder hash. Dunder len lets len of your object work, and dunder bool decides truthiness, falling back to len when absent. Dunder iter makes the object iterable in a for loop, and dunder getitem makes it indexable and sliceable. Dunder add overloads the plus operator, and dunder lt defines less than, which sorting uses. You rarely write all six comparison methods by hand; functools dot total ordering fills in the rest from a couple you provide. The philosophy is simple: builtins reach these methods through syntax, and your classes join in by implementing the same names. Lesson 10 · Idiomatic Python

The data model & dunder methods

Python has no operator+ keyword and no interface to declare. You make your objects feel native by implementing the same named methods the language already calls on its builtins.

You know how to define a class. The point of this lesson isn't that you can add behaviour — it's the realisation that len(x), a + b, for i in x, and x == y are all just syntax that calls specially named methods. Implement those methods and your class plugs straight into the language.

The one idea: syntax calls dunder methods

Many languages

// keywords / special interfaces
operator + (a, b) { ... }
class V : IComparable { ... }

Idiomatic Python

# the syntax calls a named method
len(x)   # → x.__len__()
a + b    # → a.__add__(b)

These double-underscore methods are called dunders. The builtins implement them — that's why a list already responds to len(), +, and in. Your class opts into the exact same protocol by defining the same names.

Our one class: a 2-D vector

class Vector:
    def __init__(self, x, y):
        self.x, self.y = x, y

Right now print(Vector(1, 2)) shows something like <Vector object at 0x10f...> — useless. Let's fix that first.

__repr__ vs __str__

repr() is for developers: it should be unambiguous, and ideally a string you could paste back to recreate the object. str() is the friendly form for end users. If you only write one, write __repr__ — str() falls back to it.

    def __repr__(self):
        return f"Vector({self.x!r}, {self.y!r})"
Why !r: the !r conversion calls repr() on each field, so strings keep their quotes and the result round-trips. This is the one dunder to implement on every class.

__eq__ — and pairing it with __hash__

By default two distinct objects are never equal. Define __eq__ to get value equality:

    def __eq__(self, other):
        if not isinstance(other, Vector):
            return NotImplemented
        return (self.x, self.y) == (other.x, other.y)

    def __hash__(self):
        return hash((self.x, self.y))
The trap: defining __eq__ sets __hash__ to None, making instances unhashable — they can't go in a set or be dict keys. If your value is immutable, define __hash__ too, and keep the rule: equal objects must hash equal. Returning NotImplemented (not raising) lets Python try the other operand's __eq__.

__len__ and __bool__

For a 2-D vector, "length" naturally means its dimension count, and "truthiness" means non-zero. len(obj) calls __len__; truthiness checks call __bool__, which falls back to __len__ when absent:

    def __len__(self):
        return 2

    def __bool__(self):
        return bool(self.x or self.y)

So bool(Vector(0, 0)) is False, and an if vector: test reads naturally — no len(...) != 0 in sight.

__iter__ / __getitem__ — iterable & indexable

Define __iter__ and the object works in a for loop, in unpacking, and anywhere an iterable is accepted. Define __getitem__ and it supports indexing — and slicing — too:

    def __iter__(self):
        yield self.x
        yield self.y

    def __getitem__(self, i):
        return (self.x, self.y)[i]

Now x, y = vector unpacks, vector[0] indexes, and vector[:1] slices — because indexing into a tuple already supports slices. This ties straight back to the iteration and slicing lessons.

Operator overloading: __add__ and __lt__

    def __add__(self, other):
        return Vector(self.x + other.x, self.y + other.y)

    def __lt__(self, other):
        return (self.x, self.y) < (other.x, other.y)

Now Vector(1, 2) + Vector(3, 4) gives Vector(4, 6), and a list of vectors will sort() — because sorting only needs <, i.e. __lt__.

Don't hand-write all six: the comparison operators (<, <=, >, >=, ==, !=) are tedious to define one by one. Decorate the class with @functools.total_ordering and supply just __eq__ plus one ordering method (e.g. __lt__); it fills in the rest.

The philosophy in one line: we call these methods on builtins via syntax; your classes opt in by implementing the same protocol.
