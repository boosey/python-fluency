Welcome to lesson eight: function arguments. You already know how to call functions, so this lesson is about reading and writing Python's unusually flexible signatures with confidence. Every Python argument can be passed by position or by name, and choosing names at the call site is one of the cheapest readability wins there is. When you see a bare True or a lone number in a call, the reader has to guess what it means; passing it as verbose equals True spells out the intent. Default values let a parameter fall back to something sensible when the caller stays silent. But there is a famous trap: a default value is evaluated once, when the function is defined, not on every call. So if you write def f with items defaulting to an empty list, that one list is shared across every call that relies on the default, and it quietly accumulates. The fix is to default to None, then inside the body write items equals items or None-becomes-a-fresh-list. A clean phrasing is: if items is None, set items to a new empty list. Next, the star parameter, written args, collects any extra positional arguments into a tuple. The double-star parameter, written kwargs, collects any extra keyword arguments into a dictionary. Together they let a function accept an arbitrary number of arguments, which is how wrappers and decorators stay general. The same two stars work in the other direction at the call site. Calling f of star sequence spreads a list or tuple into positional arguments, and f of double-star mapping spreads a dict into keyword arguments. A bare star in a signature is a marker: everything after it must be passed by keyword. That is how you force callers to write verbose equals True instead of a mystery positional flag. A slash in the signature does the opposite, marking the parameters before it as positional-only, which you will mostly meet in the standard library. The guiding idiom is simple: prefer keyword arguments for booleans and flags, because a name at the call site is worth a comment. Lesson 08 · Idiomatic Python

Function arguments

Python signatures are more flexible than almost any language you've used. Learn to read and write them, and you can follow real-world code — and APIs — without squinting.

You already know how to call a function. The win here is fluency with the shape of Python signatures: passing by name, safe defaults, *args/**kwargs, and call-site unpacking.

Positional or keyword — your choice

Every ordinary parameter can be filled by position or by name. Passing by name is the idiom whenever the value alone wouldn't tell the reader what it is:

Mystery positionals

# what do these mean?
connect("db", 5432, True)

Idiomatic Python

# the names are the documentation
connect("db", port=5432, ssl=True)

Default arguments — and the classic trap

A default lets a parameter fall back to a value when the caller stays silent. But a default is evaluated once, at def time — not on every call. A mutable default is therefore shared across calls and accumulates:

# BROKEN: the one list is reused on every default call
def append_to(value, items=[]):
    items.append(value)
    return items

append_to(1)  # [1]
append_to(2)  # [1, 2]  — surprise!

The fix is the idiom you'll see everywhere: default to None, then build a fresh object inside the body.

# FIXED: a new list every call
def append_to(value, items=None):
    if items is None:
        items = []
    items.append(value)
    return items
Rule of thumb: never put a mutable value ([], {}, set()) in a default. Use None as the sentinel and create the real object in the body — either if items is None: items = [] or the terser items = items or [].

*args and **kwargs: collect the extras

A * parameter collects leftover positional arguments into a tuple; a ** parameter collects leftover keyword arguments into a dict. The names are convention, not syntax — the stars do the work:

def log(level, *args, **kwargs):
    print(level, args, kwargs)

log("INFO", 1, 2, user="ada")
# INFO (1, 2) {'user': 'ada'}

This is how wrappers and decorators stay general: they accept (*args, **kwargs) and forward them on unchanged.

Unpacking at the call site

The same two stars run in reverse when calling. * spreads a sequence into positional arguments; ** spreads a mapping into keyword arguments:

point = (3, 4)
opts  = {"port": 5432, "ssl": True}

move(*point)        # same as move(3, 4)
connect("db", **opts)  # same as connect("db", port=5432, ssl=True)

Keyword-only arguments: the bare *

A bare * in a signature is a fence: every parameter after it must be passed by name. This is how a library forces callers to be explicit about flags instead of trailing a mystery boolean:

def render(template, *, verbose=False):
    ...

render("page", verbose=True)   # OK — named, readable
render("page", True)           # TypeError: too many positional args

The mirror image is the positional-only marker / (Python 3.8+): parameters before it can't be passed by name, e.g. def f(a, b, /):. You'll meet it mostly in the standard library; you'll rarely need to write it yourself.

The idiom to keep: prefer keyword arguments for booleans and flags. A bare True at a call site tells the reader nothing; verbose=True reads like a sentence. When you design a function with flags, put them after a bare * so callers can't omit the name.
