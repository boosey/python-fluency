Welcome to lesson four: slicing. Slicing is the notation Python gives you for pulling a contiguous run of items out of any sequence, and reading it fluently pays off on almost every line. The basic form is sequence, square bracket, start, colon, stop. The single rule that unlocks everything is that the interval is half-open: it includes start and excludes stop. That is why the length of the slice is simply stop minus start, with no off-by-one arithmetic to second-guess. It is also why adjacent slices like zero to n and n to the end fit together perfectly, sharing the index n with no gap and no overlap. Either bound can be omitted, and the omission means "as far as it goes": colon n is the first n items, n colon is everything from n onward. A bare colon means the whole thing, and on a list that is the idiomatic shallow copy. Negative indices count from the right, so minus one is the last item and minus three colon gives you the last three. And colon minus one gives you everything except the last item, which reads much better than length minus one. There is an optional third number, the step: colon colon two takes every second element. A step of minus one walks the sequence backwards, so sequence colon colon minus one is the idiomatic reverse. Here is the property that surprises people coming from other languages: a slice never raises IndexError. An out-of-range bound is quietly clamped to the edge, so slicing past the end just gives you what is there, even an empty sequence. That is the opposite of indexing with a single integer, which does raise when you overshoot. On a mutable sequence like a list, a slice is also an assignment target: you can replace a whole region in one statement, and the replacement need not be the same length. You can delete a region the same way with del and a slice. Strings, tuples, bytes, and ranges all slice with the identical syntax, but because they are immutable their slices return a fresh copy rather than a view. One last idea to file away: the brackets are really passing a slice object to the data model, and a colon expression just constructs one for you. Lesson 04 · Idiomatic Python

Slicing

One compact notation reads a whole region out of any sequence — master the half-open interval and slices stop being something you puzzle over and become something you skim.

You already index into arrays. The win here is reading and writing Python's slice syntax without translating it in your head — and internalising the one rule (stop is exclusive) that makes every other slice fall into place.

The half-open interval

Many languages

# compute a length, mind the +1
sub = []
for i in range(start, stop):
    sub.append(seq[i])

Idiomatic Python

# one expression; includes start, excludes stop
sub = seq[start:stop]

Read seq[2:5] as “from index 2 up to but not including 5.” The interval is half-open, and that single choice is what makes slicing pleasant: the length is just stop - start (here 5 - 2 == 3 items), and adjacent slices like seq[:n] and seq[n:] meet exactly at n — no gap, no overlap, no off-by-one.

Omit a bound to mean “all the way”

seq[:n]   # the first n items   (start defaults to 0)
seq[n:]   # everything from n on (stop defaults to len)
seq[:]    # the whole sequence — a shallow copy of a list

A missing bound means “as far as it goes.” seq[:] on a list is the idiomatic shallow copy: a new list, same elements.

Negative indices count from the right

seq[-1]    # last item
seq[-3:]   # last three items
seq[:-1]   # everything except the last item

Negative numbers index from the end, so you never write seq[len(seq) - 1] again. seq[:-1] (“all but the last”) is far more readable than computing the length yourself.

The third number: step

seq[::2]    # every second item
seq[::-1]   # the whole sequence, reversed

An optional start:stop:step third value sets the stride. A step of -1 walks backwards, so seq[::-1] is the idiomatic reverse — it works on strings and tuples too, where there is no .reverse() method to reach for.

Slices never raise IndexError. An out-of-range bound is clamped to the edge: seq[:1000] on a short list just returns what's there, and seq[1000:] returns []. Contrast a single integer index — seq[1000] — which does raise. Slicing is forgiving; indexing is strict.

Slices are assignment targets (on mutable sequences)

On a list, a slice isn't just for reading — it names a region you can replace or delete, and the replacement need not be the same length:

xs = [0, 1, 2, 3, 4]
xs[1:3] = ["a", "b", "c"]   # grow the region: [0, "a", "b", "c", 3, 4]
del xs[1:3]                # remove a region in place

Strings, tuples, bytes, and range slice with the same syntax, but they're immutable — so a slice of a string returns a brand-new string (a copy), and slice assignment on them is an error.

Peek under the hood: seq[1:3] is sugar — Python builds a slice(1, 3) object and hands it to seq.__getitem__. You can make one explicitly with slice(start, stop, step). We'll unpack that in the data-model lesson; for now, just know the colon syntax constructs an object.
