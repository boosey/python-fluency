Welcome to lesson nine: context managers. In most languages, when you open a file you must remember to close it, usually in a finally block so it still closes if an exception is thrown. Python folds that whole setup-and-teardown dance into one piece of syntax: the with statement. You write with open path as f, and the file is guaranteed to close the moment the block ends. That guarantee holds even if the body raises an exception, just like a finally clause, but without the boilerplate. So the win is deterministic cleanup tied to a block: no manual close, no finally, no leaked handle. The as part binds whatever the manager hands back, which for open is the file object itself. You can manage several resources in one statement: with open a as fa, open b as fb, and both close cleanly. Files are the famous example, but the protocol is everywhere once you start looking. A threading Lock is a context manager: with lock acquires it on entry and releases it on exit, even if the body blows up. The tempfile module gives you temporary files and directories that clean themselves up when the block ends. contextlib suppress lets you swallow a specific exception in one readable line instead of a try except that does nothing. And when you want your own manager, the easiest way is the contextlib contextmanager decorator on a generator. You write setup, then yield the value, then teardown, and the yield is where the body runs. Under the hood, any object with dunder enter and dunder exit can be a context manager, and we will meet those dunder methods properly in the next lesson. dunder enter runs on the way in and its return value is what as binds; dunder exit runs on the way out and handles any exception. Even the decimal module uses this: decimal localcontext temporarily changes precision just inside the block. The mental shift is to stop thinking close it when I am done and start thinking this resource lives exactly as long as this block. Whenever you see a try finally whose only job is cleanup, that is usually a context manager waiting to happen. Scroll down, take the quick checks, and replay this narration whenever you like. Lesson 09 · Idiomatic Python

Context managers

One block, guaranteed cleanup. with ties a resource's whole lifetime to a scope — so it's released the instant you leave, exception or not.

You already know to free what you acquire. The point of this lesson isn't that you must clean up — it's that Python has a dedicated syntax that makes the cleanup automatic and impossible to forget.

The pattern Python replaces

Most languages

# acquire, use, release in finally
f = open(path)
try:
    data = f.read()
finally:
    f.close()

Idiomatic Python

# the file closes for you
with open(path) as f:
    data = f.read()

Read it as: “open(path) for the duration of this block, called f.” When the block ends — whether it falls off the bottom, returns, or raises — the file is closed. The finally is built in.

The mental shift: stop thinking “close it when I'm done” and start thinking “this resource lives exactly as long as this block.” The with statement makes the resource's lifetime visible in the indentation.

The as binds the entry value

The thing after as is whatever the manager hands back on entry — for open that's the file object, but it isn't always the same object you called with on. A manager can return anything useful:

with open(path) as f:        # f is the file object
    for line in f:
        process(line)

You can also use a manager with no as at all when you only care about its side effect (a lock, a timer, a redirect).

Managing several resources at once

# both files close, in reverse order, no matter what
with open(src) as fa, open(dst, "w") as fb:
    fb.write(fa.read())

Comma-separate the managers in one with. They're entered left-to-right and exited right-to-left, so fb closes before fa. (For long lists, parenthesise across lines on 3.10+.)

Managers you'll meet constantly

# a lock: acquired on entry, released on exit — even on error
with lock:
    counter += 1

# a self-cleaning temp directory
import tempfile
with tempfile.TemporaryDirectory() as d:
    work_in(d)            # d and its contents vanish at block end

# swallow exactly one expected error, no empty except
from contextlib import suppress
with suppress(FileNotFoundError):
    os.remove(stale)

threading.Lock, tempfile, and contextlib.suppress all speak the same protocol as open. Once you recognise the shape, with reads as “do this safely, then tidy up.”

Writing your own — the easy way

You rarely need a class. contextlib.contextmanager turns a generator into a manager: everything before yield is setup, everything after is teardown, and yield is where the body runs.

from contextlib import contextmanager

@contextmanager
def timed(label):
    start = time.perf_counter()
    try:
        yield                       # the with-block runs here
    finally:
        dt = time.perf_counter() - start
        print(f"{label}: {dt:.3f}s")

with timed("load"):
    rows = load()

The finally around the yield guarantees teardown even if the block raises — the same safety open gives you, now in your own code. Whatever you yield becomes the as value.

The protocol underneath

Any object is a context manager if it defines two methods: __enter__ runs on the way in and its return value is what as binds; __exit__ runs on the way out and receives any exception so it can clean up (and, if it chooses, suppress it).

These are dunder methods — part of Python's data model. We meet them properly in the next lesson; for now, just know with calls them for you. Even decimal.localcontext() uses this to restore precision when its block ends.
