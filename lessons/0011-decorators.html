The whole idea rests on one fact: in Python, functions are first-class objects. You can assign a function to a variable, pass it into another function, store it in a list, and return it from a function, just like any other value. Once you accept that, a decorator is almost boring: it is just a function that takes a function and returns another function. The at-sign line is pure sugar. Writing at-timer above a function definition is exactly the same as writing func equals timer of func right after it. Nothing more is happening; the name is rebound to whatever the decorator returns. Let us build one from scratch so it stops being mysterious. Inside the decorator we define a small wrapper function that accepts star-args and star-star-kwargs, calls the original, and adds behaviour around it. Lesson 11 · Idiomatic Python

Decorators

The @ line that sits above so many Python functions isn't magic — it's one tiny rule applied consistently. Learn the rule and you read and write decorators with confidence.

You'll write your own decorators only occasionally, but you'll read them constantly — in web frameworks, in test suites, in the standard library. The win for this lesson is decoding any @something on sight.

The foundation: functions are first-class

Before decorators make sense, one fact has to be automatic: a function in Python is just an object. You can assign it, pass it, return it, and store it — no calling parentheses required.

def shout(text):
    return text.upper()

greet = shout              # assign the function itself (no parens)
greet("hi")               # -> "HI"

funcs = [shout, str.strip]  # store functions in a list
for f in funcs:
    f("  hi  ")            # call each one
The mental shift: shout is a value; shout() is the act of calling it. A decorator is built entirely on passing the value around before anyone calls it.

What a decorator actually is

A decorator is a function that takes a function and returns a function — usually a new one that wraps the original with extra behaviour. The @ syntax is pure sugar for a reassignment:

The desugared form

def greet():
    return "hi"

greet = timer(greet)

Idiomatic Python

@timer
def greet():
    return "hi"

Both lines mean the identical thing: “call timer with greet, and rebind the name greet to whatever comes back.” Read every @decorator as name = decorator(name) and the mystery is gone.

Build one from scratch

Here is a complete, idiomatic timer. The pattern is always the same: define a wrapper inside, do something before and after, and return the wrapper.

import functools, time

def timer(func):
    @functools.wraps(func)
    def wrapper(*args, **kwargs):
        start = time.perf_counter()
        result = func(*args, **kwargs)     # call the real function
        elapsed = time.perf_counter() - start
        print(f"{func.__name__} took {elapsed:.4f}s")
        return result                      # pass its result back through
    return wrapper                         # return the replacement

@timer
def slow_add(a, b):
    time.sleep(0.1)
    return a + b
Why *args, **kwargs? The wrapper stands in for any function, so it must accept any arguments and forward them unchanged. Capturing with *args, **kwargs and re-passing them to func makes one decorator work on every signature.

The tax: functools.wraps

Because the wrapper replaces the original, the decorated function starts reporting the wrapper's identity — its __name__ becomes "wrapper" and its __doc__ is lost. That breaks help(), debuggers, and doc tools.

Without @functools.wraps

slow_add.__name__
# -> 'wrapper'  (wrong!)

With @functools.wraps(func)

slow_add.__name__
# -> 'slow_add'  (preserved)

functools.wraps(func) copies __name__, __doc__, and friends from the original onto your wrapper. It costs one line — always include it.

The ones you'll meet in the wild

You rarely hand-write decorators, but the standard library hands you several you'll use constantly:

@functools.cache             # memoise: cache results by arguments (3.9+)
def fib(n):
    return n if n < 2 else fib(n - 1) + fib(n - 2)

class Circle:
    def __init__(self, r):
        self.r = r

    @property                  # call-free attribute access: c.area
    def area(self):
        return 3.14159 * self.r ** 2

    @staticmethod              # no self / no cls
    def describe():
        return "a round shape"

    @classmethod               # receives the class as cls
    def unit(cls):
        return cls(1)

You'll also meet @functools.lru_cache(maxsize=...) (the bounded-size sibling of @cache) and @dataclass, which generates __init__ and __repr__ for a class — we give it its own lesson later.

Decorators that take arguments

When you see @retry(times=3), that () means an extra layer. A parameterised decorator is a decorator factory: an outer function that takes the arguments and returns the real decorator.

def repeat(times):              # factory: takes the config...
    def decorator(func):         # ...returns a real decorator...
        @functools.wraps(func)
        def wrapper(*args, **kwargs):
            for _ in range(times):
                result = func(*args, **kwargs)
            return result
        return wrapper
    return decorator

@repeat(times=3)               # repeat(3) returns the decorator
def ping():
    print("ping")

@repeat(times=3) first calls repeat(times=3), which returns decorator; that decorator is then applied to ping. Three nestings instead of two — same rule each time. Keep this in your back pocket; don't reach for it unless you genuinely need the argument.
