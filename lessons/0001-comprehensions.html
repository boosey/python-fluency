Welcome to lesson one: comprehensions. This is the single most recognisable piece of Python syntax, so it is the fastest win toward reading Python fluently. In most languages, to build a new list from an old one, you create an empty list, write a loop, and append inside it. Python collapses that whole pattern into one expression called a list comprehension. You read it left to right as: the thing I want, for each item, from this source, optionally where a condition holds. So, square brackets, n times n, for n in numbers, gives you a list of squares. Add an if clause at the end to filter: n times n, for n in numbers, if n is even. The key mental shift is that a comprehension is an expression, not a statement. It evaluates to a value, so you can pass it straight to a function or return it directly. The same shape works for dictionaries and sets: swap the square brackets for curly braces. Curly braces with a key colon value pair build a dict; curly braces with single values build a set. Now the most important variant for an experienced programmer: the generator expression. Swap the square brackets for round brackets and nothing is built in memory. Instead you get a lazy iterator that produces one value at a time, only as it is consumed. That means sum of n times n for n in a billion numbers uses almost no memory, because values flow through one at a time. When you pass a generator expression as the only argument to a function, you can even drop the extra parentheses. A word of restraint: comprehensions are for building a collection from an expression. If the body has side effects, or you are nesting two or more loops with conditions, a plain for loop reads better. The Zen of Python says flat is better than nested, and readability counts. Lesson 01 · Idiomatic Python

Comprehensions

The most recognisable line of Python you'll ever read — and the fastest way to start writing Python that looks native instead of ported.

You already know how to transform a collection. The point of this lesson is not that you can — it's to make Python's one-line form so automatic that you read and write it without slowing down.

The pattern Python replaces

Most languages

# build, loop, append
squares = []
for n in numbers:
    squares.append(n * n)

Idiomatic Python

# one expression
squares = [n * n for n in numbers]

Read it left-to-right as a sentence: “n * n — for each n in numbers.” The expression comes first because the result is what you care about, not the mechanics.

Filtering: the trailing if

evens   = [n      for n in numbers if n % 2 == 0]
labels  = [f"#{n}" for n in numbers if n > 0]

A trailing if selects which items make it through. (An if/else placed before the for is a different thing — a conditional expression that transforms each item — a later lesson.)

The mental shift: a comprehension is an expression, not a statement. It evaluates to a value, so you can return it, pass it to a function, or nest it directly — no temporary variable required.

Same shape, three containers

# list  — square brackets
[n * n for n in nums]

# set   — curly braces, single values (deduplicates)
{n % 10 for n in nums}

# dict  — curly braces, key: value pairs
{n: n * n for n in nums}

The one that matters most: generator expressions

Swap the square brackets for round ones and nothing is built in memory. You get a lazy iterator that yields one value at a time, as it's consumed:

# list comp: builds a million-item list in memory
total = sum([n * n for n in range(1_000_000)])

# generator: values flow through one at a time — near-zero memory
total = sum(n * n for n in range(1_000_000))

When the generator expression is the only argument to a function, you drop the extra parentheses — as above. This is everyday idiomatic Python: reach for a generator when you're feeding a pipeline (sum, any, max, "".join(...)) rather than keeping the whole collection.

When not to use one

Comprehensions are for building a collection from an expression. Step back to a plain for loop when:

  • the body has side effects (logging, writing, mutating) — a comprehension built only for its side effects is a code smell;
  • you'd nest two or more for/if clauses and it stops reading like a sentence.

“Flat is better than nested. Readability counts.” — the Zen of Python (PEP 20).
