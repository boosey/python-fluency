Welcome to lesson two: unpacking and multiple assignment. This is the idiom that lets you pull values out of sequences in a single readable line, instead of indexing one element at a time. The starting point is multiple assignment: a comma b equals one comma two binds two names at once. The right side is really a tuple, and Python takes it apart to match the names on the left. The famous payoff is swapping two variables: a comma b equals b comma a, with no temporary variable at all. Python evaluates the whole right side first, then assigns, so the swap just works. Unpacking is not limited to tuples; it works on any iterable, including lists, strings, and the results of a function call. You will see it most often in for loops: for key comma value in pairs reads the two parts of each item directly. When a sequence is longer than the names, star unpacking captures the rest into a list. First comma star rest equals the sequence puts the head in first and everything else in rest. You can put the star in the middle or at the end too: star init comma last, or head comma star mid comma tail. The starred name always becomes a list, even when it captures nothing. When you do not care about a value, bind it to a single underscore by convention. A starred underscore, star underscore, swallows several values you want to ignore. Unpacking nests, so a comma open paren b comma c close paren mirrors the shape of the data. The same star and double-star work in the other direction, when you build calls and literals. A single star spreads an iterable into positional arguments or into a new list or tuple. A double star spreads a dictionary into keyword arguments, or merges dicts inside curly braces. Extended unpacking with the star came from PEP three one three two, and the dict union operator pipe arrived in Python three point nine. Lesson 02 · Idiomatic Python

Unpacking & multiple assignment

Stop indexing sequences one element at a time. Python lets you name the pieces in a single line — and once it's automatic, half your “glue” code disappears.

You already know how to read pair[0] and pair[1]. The point of this lesson isn't that you can — it's to make Python's unpacking forms so reflexive that you stop writing the index at all.

Multiple assignment, and the swap

Most languages

# swap needs a temp
tmp = a
a = b
b = tmp

Idiomatic Python

# one line, no temp
a, b = b, a

The right-hand side b, a is a tuple. Python evaluates that whole tuple first, then binds the names on the left — which is exactly why the swap needs no temporary. The same mechanism powers plain multiple assignment: a, b = 1, 2.

Unpacking any iterable

The left side describes a shape; the right side is any iterable that matches it. Lengths must agree or you get a ValueError.

x, y, z = [1, 2, 3]       # from a list
r, g, b = "RGB"           # from a string
name, age = get_user()      # from a returned tuple
The mental shift: unpacking isn't a special tuple feature — it's how assignment itself works against any iterable. Once you see that, the for-loop form below stops looking like magic.

Unpacking in for loops

This is where you'll meet it most. Each item is unpacked as the loop binds it:

for k, v in pairs:              # pairs of (key, value)
    print(k, "->", v)

for i, item in enumerate(seq):  # index + value, idiomatic
    print(i, item)

Star unpacking: capture the rest

When the iterable is longer than your names, a single starred name absorbs whatever's left — as a list, always, even if it's empty:

first, *rest  = [1, 2, 3, 4]   # first=1,  rest=[2, 3, 4]
*init, last   = [1, 2, 3, 4]   # init=[1, 2, 3],  last=4
head, *mid, tail = [1, 2, 3, 4] # head=1, mid=[2, 3], tail=4

Exactly one star is allowed per target list — Python needs an unambiguous pivot point. This is PEP 3132 “extended iterable unpacking”.

Ignoring values

Bind throwaway values to a single underscore _ by convention, and use *_ to swallow several you don't care about:

_, y = point              # ignore the x coordinate
first, *_ = record       # keep the head, drop the rest
name, *_, country = row  # keep the ends, drop the middle

Nested unpacking

The target can mirror nested structure — parentheses on the left match parentheses (or any grouping) on the right:

(a, (b, c)) = (1, (2, 3))         # a=1, b=2, c=3

for name, (lat, lon) in cities:    # unpack the pair inside each item
    print(name, lat, lon)

The other direction: * and ** in calls and literals

The same stars spread values when you build things. A single * spreads an iterable; a double ** spreads a dict:

print(*args)                # spread an iterable as positional args
merged_list  = [*a, *b]      # concatenate into a new list
merged_tuple = (*a, *b)      # ...or a new tuple

config = {**base, "k": 1}    # copy base, then override "k"
merged = {**d1, **d2}        # later keys win on conflict
greet(**kwargs)              # spread a dict as keyword args
Two ways to merge dicts: the {**d1, **d2} form works everywhere, and on conflict the later key wins. Python 3.9+ also has the union operator d1 | d2, which reads more cleanly — but {**d1, **d2} is the form you'll see in code targeting any 3.x.

When to reach for it

The rule of thumb is simple: whenever you're about to write an index, ask whether unpacking reads better. It usually does. Step back only when the shape is genuinely dynamic — when you don't know how many elements you'll get and can't express that with a single star.
